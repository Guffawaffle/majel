## Scope

Defensive security review of Majel runtime surfaces with emphasis on:
- dependency vulnerabilities,
- import/upload entry points (`csv`/`tsv`/`xlsx`, image payloads),
- translator config and payload validation,
- practical hardening suitable for portfolio demonstration.

## What triggered this sweep

A Copilot-side safety-filtered request produced a UI `Server error: 500`.
This was confirmed to be a tooling/request-pipeline failure, not an app runtime 500 from Majel.

## Findings

### 1) Dependency vulnerabilities (HIGH x2)

Initial `npm audit --json` findings:
- `minimatch` ReDoS (`GHSA-7r86-cg39-jmmj`, `GHSA-23c5-xmqv-rm74`)
- `rollup` arbitrary file write/path traversal (`GHSA-mw96-cpmx-2vgc`)

Root cause: transitive dependency ranges in toolchain graph.

### 2) Import pipeline hardening opportunities

Observed risk patterns:
- accepted `contentBase64` without strict base64 format validation,
- no decoded-size check before parsing,
- parser accepted unbounded row/column/cell sizes (practical DoS risk via large structured files).

### 3) Translator route input hardening opportunities

Observed risk patterns:
- `configName` checks prevented separators but allowed broader input than needed,
- payload accepted any object-ish value (including arrays),
- `dry_run` did not enforce boolean type.

### Import security guards
Files:
- `src/server/services/route-helpers/imports-helpers.ts`
- `src/server/services/import-mapping.ts`

Implemented:
- strict base64 regex validation,
- decoded-byte estimate cap check,
- parser safety limits:
  - max rows: header + 10,000 data rows,
  - max columns: 200,
  - max cell length: 20,000 chars,
- enforcement applied to both delimited and xlsx parse paths.

### Translator route guards
File:
- `src/server/routes/translator.ts`

Implemented:
- strict `configName` allowlist regex (`^[a-zA-Z0-9_-]{1,80}$`),
- non-array object requirement for `payload`,
- `dry_run` type validation (must be boolean when provided).

### Regression tests

Added tests for:
